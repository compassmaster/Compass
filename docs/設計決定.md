# 設計決定（Architecture Decision Records）

Compassの設計に関する重要な意思決定を記録する。

本ドキュメントに記載されたAcceptedのDecisionは、Compassにおける正式な設計方針として扱う。

---

## テンプレート

新しいDecisionを追加する際は、以下のフォーマットを使用し、`## Decisions`の直下に追加する。

### D-[番号]: [タイトル]

- **日付**: YYYY-MM-DD
- **ステータス**: Proposed / Accepted / Rejected / Deprecated
- **背景**: なぜこの決定が必要になったのか
- **決定内容**: 何を決定したのか
- **理由**: なぜそのように決定したのか
- **影響範囲**: この決定によって影響を受けるコンポーネントやフロー

## Decisions

### D-0007: EvidenceからUnderstanding Candidateを生成し、ユーザー確認前にUserModelへ反映しない

- **日付**: 2026-07-22
- **ステータス**: Accepted
- **背景**:  
  D-0002では、User Modelを根拠付きの仮説として管理することを定義した。D-0003では、その理解をユーザーと共同で検証する体験を設計し、D-0005ではImmediate ResponseとReflectionを分離した。しかし、Analysisが生成したEvidenceをどのようにUser Modelへつなげるか、その中間段階はまだ明確に定義されていなかった。

  Analysisの役割は、観測された事実をEvidenceとして整理することであり、人格や価値観を解釈することではない。一方で、EvidenceをそのままUser Modelへ反映すると、ユーザーの確認を経ずにAIが人格や傾向を決めつけてしまう可能性がある。

  Compassが「AIがユーザーを決める」のではなく、「AIとユーザーが一緒に理解を育てる」存在であるためには、EvidenceとUser Modelの間に、共同で検証するための中間レイヤーが必要となった。

- **決定内容**:

  Compassでは、EvidenceとUser Modelの間に **Understanding Candidate** を設ける。

  Understanding Candidateは、Analysisによって生成されたEvidenceを、人間が確認可能な「理解候補（仮説）」へ変換したものである。

  Understanding Candidateは、以下の原則を満たす。

  - 必ず1件以上のEvidenceを根拠として保持する
  - Evidenceが存在しないCandidateを生成してはならない
  - Candidateは「観測された可能性」として扱い、断定的な表現を使用しない
  - Candidateは必ずユーザーによる確認を経る
  - ユーザーは「そう思う」「少し違う」「まだ分からない」などのフィードバックを行える
  - ユーザー確認前のCandidateをUser Modelへ反映してはならない
  - Candidateの生成と、User Modelへの反映は別責務として実装する

  Compassの理解フローを以下のように定義する。

  ```text
  Daily Log
        │
        ▼
  Analysis
        │
        ▼
  Evidence
        │
        ▼
  Understanding Candidate
        │
        ▼
  User Confirmation
        │
        ▼
  User Model
        │
        ▼
  Compass Map
  ```

  Understanding Candidateは、AIが一方的に決定する人格診断ではなく、AIとユーザーが共同で検証するための一時的な理解候補として扱う。

- **理由**:  
  Analysis・Understanding・User Modelの責務を明確に分離することで、観測された事実とAIによる解釈を混同しない設計を実現するためである。また、AIがユーザーを勝手にラベリングすることを防ぎ、理解の透明性と修正可能性を維持できる。

  Evidenceを経由せずUser Modelを更新できない構造とすることで、Compassの「根拠のある理解」という設計思想をシステムレベルで保証できる。

- **影響範囲**:
  - Analysis Service
  - Evidence
  - Understanding Candidate
  - Candidate Generator
  - User Confirmation UI
  - User Model更新フロー
  - Reflection
  - Compass Map
  - 将来のLLMによる理解生成
  
### D-0006: 睡眠情報を日単位で管理し、就寝・起床時刻を記録する

- **日付**: 2026-07-21
- **ステータス**: Accepted
- **背景**: 現在のDaily Logでは、睡眠時間を各記録に含めている。そのため、朝に睡眠時間を入力した後、昼や夜に追加記録する際にも同じ情報を再入力する必要があり、継続利用時の負担になっている。また、合計睡眠時間のみを1時間単位で記録しているため、6時間30分などの睡眠を直感的に入力しにくい。就寝時刻・起床時刻も保存していないため、生活リズムの変動を分析できない。将来的なスマートウォッチ連携や生活リズム分析に対応できるデータ構造が必要になった。
- **決定内容**:

  睡眠情報を各Daily Logに直接保持する方式から、日単位で1件のSleep Recordとして管理する方式へ移行する。

  Sleep Recordは、少なくとも以下の情報を持つ。

  - 対象日
  - 就寝日時
  - 起床日時
  - 睡眠時間
  - 入力元
  - 作成日時
  - 更新日時

  睡眠時間は、就寝日時と起床日時から分単位で自動計算する。

  Sleep Recordの対象日は、原則として起床した日とする。

  同日に複数回Daily Logを記録しても、睡眠情報は再入力しない。入力ミスや記録漏れがある場合、ユーザーは後からその日のSleep Recordを編集できるようにする。

  睡眠情報を編集した場合は、更新後の情報を使用して関連する分析結果を再計算できる設計とする。

  将来的には、スマートウォッチなどから取得した元データと、ユーザーが確認・修正した確定値を区別して保持できる構造へ拡張する。

  生活リズム分析では、睡眠時間だけでなく、以下の情報を分析対象とする。

  - 就寝時刻
  - 起床時刻
  - 就寝・起床時刻のばらつき
  - 曜日による違い
  - 平日と休日の差
  - 連続した生活リズムの変化
  - 睡眠と気分・疲労・集中状態との関係

  将来的な分析エンジンでは、あらかじめ定義された分析だけでなく、AIが分析候補を提案し、統計的な検証とユーザーフィードバックを通して、そのユーザーにとって有効な分析方法を選択・改善する仕組みを目指す。

- **理由**: 睡眠は記録した瞬間ごとに変化する情報ではなく、基本的にその日全体に対応する情報である。そのため、Daily Logごとに保持するよりも、日単位の独立したデータとして管理する方が責務として自然である。就寝・起床時刻から睡眠時間を自動計算することで、入力負担と入力誤差を減らし、生活リズム分析に必要な情報を蓄積できる。また、後から編集可能にすることで、手動入力の誤りやスマートウォッチによる推定誤差も補正できる。
- **影響範囲**: Daily Logのデータモデル、Sleep Recordの型・Repository・Application Service、記録画面、睡眠編集画面、既存データのマイグレーション、Analysis Service、Sleep Analyzer、Fatigue Analyzer、Reflection、Compass Map、将来のスマートウォッチ連携および機械学習基盤。

### D-0005: Compassの理解処理に二段階モデルを採用する

- **日付**: 2026-07-16
- **ステータス**: Accepted
- **背景**: ユーザーがDaily Logを保存した直後に、AIが過去のデータを含む深い分析や人格理解まで行うと、1日の断片的な情報から誤った判断を行う可能性がある。一方で、保存直後に何の反応もない場合、ユーザーは記録が受け止められたという安心感を得にくい。そのため、保存直後の応答と、蓄積データを用いた深い理解を分離する必要があった。
- **決定内容**:

  Compassは、以下の2つの処理を組み合わせたハイブリッドアーキテクチャを採用する。

  #### 1. Immediate Response（即時反応）

  - **タイミング**: ユーザーがDaily Logを保存した直後
  - **役割**: 記録の受信と安心感の提供
  - **処理内容**:
    - ログが正常に保存されたことを伝える
    - その日の感情や出来事に対して、浅く温かい傾聴と受容を行う
    - 深い人格分析を行わない
    - User Modelを更新しない
    - 1件のログだけから長期的な理解を生成しない

  #### 2. Reflection（深い理解）

  - **タイミング**: 非同期処理、夜間バッチ、または一定期間のデータ蓄積後
  - **役割**: パターンの発見とユーザー理解の更新
  - **処理内容**:
    - 過去のログ、環境データ、睡眠、行動パターンなどを総合的に分析する
    - 単発の出来事ではなく、継続して観測されたパターンを扱う
    - Evidenceに基づいてUnderstanding Candidateを生成する
    - 適切なタイミングでユーザーに確認を求める
    - ユーザーの確認を経てUser Modelへ反映する

- **理由**: 人間を深く理解するには、1日の出来事だけでなく、時間の経過と複数の記録から見えるパターンを確認する必要がある。即時反応と深い理解を分離することで、ユーザーに安心感を提供しながら、断片的な情報による誤ったラベリングを防止できる。また、一つの波だけでなく、風や潮流、過去の航路から進路を考える「航海士」というCompassの世界観にも合致する。
- **影響範囲**: Daily Log保存フロー、Immediate Response生成処理、Analysis Service、Reflection、Understanding Candidate生成、User Model更新フロー、夜間バッチ処理、会話UI。

### D-0004: Compass Analysis Personaの基本思想と5原則を定義する

- **日付**: 2026-07-16
- **ステータス**: Accepted
- **背景**: Compassがユーザーのログを分析する際、一般的な評価・診断・改善指導の考え方をそのまま適用すると、ユーザーに「監視されている」「評価されている」「決めつけられている」という感覚を与える可能性がある。Compassを「人生を共に歩むAI航海士」として成立させるためには、分析エンジンが守る姿勢を明文化する必要があった。
- **決定内容**:

  Compassの分析エンジンは、ユーザーを評価する存在ではなく、「静かな観察者であり、理解者」として振る舞う。

  ログ分析において、以下の5原則を適用する。

  #### 1. 判断より理解を優先する

  ユーザーの行動や状態に対して、良い・悪いという評価を下さない。

  「なぜその状態になったのか」「その背景に何があるのか」という理解を優先する。

  #### 2. ネガティブな状態を直ちに問題と決めつけない

  疲労、落ち込み、やる気の低下などを、直ちに改善すべき欠点として扱わない。

  それらが回復の必要性や、自分を守るための反応である可能性も考慮する。

  #### 3. 単発の行動ではなくパターンを見る

  1日の出来事だけでユーザーを判断しない。

  複数の記録、時間的な変化、環境や体調との関係から、繰り返し観測されるパターンを探す。

  #### 4. 強みと資源を発見する

  課題や問題点だけでなく、ユーザーが持つ強み、支えとなっているもの、大切にしている価値観を探索する。

  #### 5. すべての理解を仮説として扱う

  「あなたは〇〇な人です」と断定しない。

  「〇〇という傾向があるかもしれません」など、修正可能な仮説として提示する。

  最終的な判断と決定権はユーザーに委ねる。

- **理由**: Compassを単なる自己管理アプリではなく、本当に自分を理解しようとしてくれる伴走者として機能させるためである。評価や監視による不気味さを避け、心理的安全性と信頼関係を育てるには、AIの分析姿勢をシステムレベルで明確に定義する必要がある。
- **影響範囲**: Analysis Service、Analyzer、Reflection、Understanding Candidate生成、LLMのシステムプロンプト、会話生成、Compass Map、User Model更新フロー。

### D-0003: User Modelの提示・検証体験を3層構造で設計する

- **日付**: 2026-07-16
- **ステータス**: Accepted
- **背景**: D-0002で設計したUser Modelを完全なブラックボックスとして扱うと、Compassの「根拠を説明する」という原則に反し、ユーザーの不信感につながる。一方で、AIが理解した性格や価値観を常時画面に表示すると、ユーザーにラベリングされているような圧迫感を与える可能性がある。そのため、User Modelをどのような頻度と方法でユーザーに提示するかを設計する必要があった。
- **決定内容**:

  User Modelの提示と検証を、以下の3層構造として設計する。

  #### 1. Invisible Understanding（裏側で働く理解）

  - AIの推論や会話の文脈として、バックグラウンドで利用する
  - 提案内容や話し方のパーソナライズに活用する
  - ユーザーへ毎回明示的に表示しない
  - 根拠のない理解や未確認の仮説を、確定事項として使用しない

  #### 2. Reflection（共同検証）

  - 一定期間の経過後や、理解候補が十分に育ったタイミングで提示する
  - 「最近、〇〇という傾向が見えてきましたが、あなた自身はどう感じますか」のように確認する
  - ユーザーが以下のようなフィードバックを行えるようにする
    - そう思う
    - 少し違う
    - まだ分からない
    - 修正する
  - フィードバックを理解の確信度や内容の更新に反映する

  #### 3. Compass Map（ユーザーが見に行く理解の地図）

  - ユーザー自身の取扱説明書として表示する
  - 固定的な人格診断ではなく、AIとユーザーが共同で更新する航海図として扱う
  - 理解内容、確信度、根拠を確認できるようにする
  - ユーザー自身が理解を修正・削除できるようにする

- **理由**: User Modelを「AIが勝手に作るユーザー像」ではなく、「AIとユーザーが共同で育てる自己理解の地図」として位置付けるためである。表示頻度と用途を分離することで、理解を会話に活用しながら、押し付けがましさを抑え、透明性とユーザーのコントロール権を両立できる。
- **影響範囲**: Compass Map、Reflectionカード、会話UI、User Model表示処理、Understanding Candidate確認フロー、理解の修正・削除機能。

### D-0002: User Modelを長期・短期の二層構造と根拠付き仮説で管理する

- **日付**: 2026-07-15
- **ステータス**: Accepted
- **背景**: Compassは、単に会話やDaily Logを保存するだけでなく、ユーザー理解を継続的に更新する必要がある。しかし、長期的な価値観や目標と、短期的な気分や悩みを同じ性質のデータとして扱うと、一時的な状態を人格として固定してしまう可能性がある。また、AIが過去の発言や記録からユーザーの性格を決めつける「ハルシネーションの固定化」を防ぐため、理解を根拠付きの仮説として管理する必要があった。
- **決定内容**:

  User Modelの理解内容を、以下の2層に分離する。

  #### 1. Long-term Layer（長期・静的レイヤー）

  比較的長期間にわたって維持される可能性がある情報を扱う。

  - 価値観
  - 長期目標
  - 性格傾向
  - 大切にしているもの
  - 継続的に見られる行動傾向

  #### 2. Short-term Layer（短期・動的レイヤー）

  現在または直近の状態を扱う。

  - 現在の気分
  - 直近の悩み
  - 最近の興味
  - 一時的な疲労や不安
  - 現在置かれている状況

  各レイヤーの理解内容は、単なる文字列ではなく、根拠と確信度を持つ仮説として管理する。

  ```typescript
  interface UserModel {
    id: string;

    longTerm: {
      coreValues: Hypothesis<string[]>;
      longTermGoals: Hypothesis<string[]>;
      personalityTraits: Hypothesis<string[]>;
    };

    shortTerm: {
      currentMood: {
        status: string;
        intensity: number;
        lastUpdated: string;
      };

      immediateConcerns: Hypothesis<string[]>;
      recentInterests: Hypothesis<string[]>;
    };
  }

  interface Hypothesis<T> {
    value: T;
    confidence: number;
    evidenceList: Array<{
      logId: string;
      extractedText: string;
      timestamp: string;
    }>;
    lastUpdated: string;
  }
  confidenceは0.0から1.0の範囲で扱う。

Evidenceが存在しない理解を、高い確信度を持つ確定的な理解として表示・保存してはならない。

理由: 長期的な本質と短期的な状態を分離することで、一時的な感情や状況を人格として固定することを防げる。また、すべての理解を根拠付きの仮説として扱うことで、Compassがユーザーを固定的にラベリングせず、変化し続ける一人の人間として理解できる。
影響範囲: User Modelの型定義、HypothesisおよびEvidenceの型、Analysis、Understanding、Reflection、Memory、Compass Map、User Model更新フロー。
D-0001: ドキュメントの日本語化とDecision管理を導入する
日付: 2026-07-15
ステータス: Accepted
背景: Compassの設計と議論は主に日本語で行われている。現段階では国際化よりも、FounderとAIが迷わず理解できること、設計内容を素早く確認できること、開発効率を高めることを優先する必要があった。また、設計上の重要な判断が会話の中だけに残ると、後から決定理由を確認できなくなる問題があった。

決定内容:

docs/配下の主要ドキュメントについて、内容と用途が分かりやすい日本語名を使用する。

設計上の重要な意思決定は、本ファイルにArchitecture Decision Recordとして記録する。

Compassの基本的な開発フローを、以下の順序とする。

アイデア
    ↓
議論
    ↓
レビュー
    ↓
Decisionへの記録
    ↓
関連ドキュメントの更新
    ↓
実装
    ↓
テスト・レビュー
    ↓
変更履歴の更新
理由: ドキュメントを人間とAIの双方が参照できる正式な設計資産として育てるためである。また、決定内容だけでなく、その背景や理由を残すことで、将来の設計変更やレビューを行いやすくする。OSS化や英語化については、Compassの基本設計が安定した後に改めて検討する。
影響範囲: docs/配下のドキュメント、ファイル命名規則、AI協働プロトコル、AI Context、リポジトリガイド、変更履歴、開発フロー。
