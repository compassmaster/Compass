## 2026-07-23 - D-0010 External ContextとWeather Record保存境界

### 追加

* `docs/設計決定.md` にD-0010「External ContextとWeather Recordの保存境界」をAccepted ADRとして追加した。
* WeatherForecastSnapshotとObservedWeatherRecordを分離し、Location / Privacy、Missing / API failure、Prediction Snapshot、Formal Pipelineとの境界を明記した。

### 理由

* Weather Type、Repository、API Client、Analyzer、Predictionを実装する前に、予測時点の再現性、実測との比較、位置情報プライバシー、欠損処理の保存境界を確定するため。

### 未実装として維持

* Weather TypeScript型、Repository、localStorage key、API Client、Open-Meteo通信、GPS、Location UI、Weather Analyzer、Prediction、Machine Learning、Conversation、Formal UserModel変更は実装していない。
* 次の実装対象はWeather Domain Modelである。

## 2026-07-22 - Understanding Object生成境界の設計

### 何を変更したか
* D-0008として、Candidate ResponseからUnderstanding Objectを生成する境界をAccepted ADRに追加した。
* AGREEのみをUnderstanding Object生成対象とし、PARTIALLY_DISAGREE / UNSUREはObject生成なしと定義した。
* AGREEはUnderstanding Status Confirmedを意味せず、新規ObjectはHypothesis maturityから始まると明記した。
* Sleep Fatigue Understandingの初期マッピングを `SLEEP_FATIGUE_RELATIONSHIP` / `LONG_TERM` / `INTERNAL_STATE + BEHAVIOR` として設計した。
* Patternsを正式Understanding Categoryから分離し、PatternはAnalysisが見つける観測・関係の形式として整理した。
* confidenceを「現在のEvidenceがUnderstandingをどの程度支持しているか」と定義した。
* アプリケーションコードは変更していない。

### なぜ変更したか
* D-0007とUnderstanding Candidate MVPの次段階として、ユーザー回答済みCandidateをいつ正式なUnderstanding Objectへ変換するかを明確にするため。
* ユーザー同意とUnderstanding成熟度を混同せず、既存Hypothesis型UserModelから段階移行するため。

### 他への影響
* 実装済み範囲はEvidence、Understanding Candidate、Candidate Response保存までである。
* 設計済み範囲はCandidate Response → Understanding Objectである。
* Understanding Object型、Object生成処理、Object Repository、UserModel新構造、Compass Map正式反映は未実装である。

## 2026-07-15 - Compass設計思想の更新（「User Modelを育てるAI」への転換）

### 何を変更したか
* Compassのコア思想を「Memoryを持つAI」から「User Modelを育てるAI」へと大きく変更
* データの概念フローを再定義（Conversation -> Info Extraction -> Memory -> Hypothesis -> User Model -> Reasoning -> Conversation/Support/Planning）
* `docs/philosophy` ディレクトリを新設し、`Compass_Core_Philosophy.md` の枠組みを用意
* `AI_CONTEXT.md`, `CURRENT_STATE.md` を新思想に合わせて更新

### なぜ変更したか
* 議論の結果、Compassの本質は単なる記録（Memory）ではなく、「人を理解し、その理解を使って未来をより良くすること」であると結論付けられたため
* Understandingを単なる記憶の集合ではなく「人物モデル」として、PredictionをPlanning以外にも広く活用する設計にするため

### 他への影響
* 実装上のデータ構造（User Model, Memory, Understanding）の持ち方が根本から変わる可能性が高い
* 今後の全ドキュメント・実装は、この「User Modelを中心とした人物理解」のパラダイムを前提として行われる

### 今後必要になる作業
* Compass Core Philosophy の本文執筆
* 新しい概念フローに基づくアーキテクチャ設計・詳細仕様の策定
