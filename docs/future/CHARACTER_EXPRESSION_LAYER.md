# Character Expression Layer

## Purpose

Character Expression Layerは、Compassが観測・理解した内容を、ユーザーにとって親しみやすく、愛着を持ちやすく、共有しやすい形で表現するための将来構想である。

Compassの本体は、Evidence・Reflection・Understanding Candidate・UserModelを中心とした、根拠付きで慎重な理解システムである。

しかし、そのままでは体験がやや硬く、ユーザーが直感的に楽しめる要素や、他者と共有しやすい魅力が不足する可能性がある。

そのため、Compassでは将来的に、理解の本体とは別に、理解内容を「キャラクター」という形で表現する層を導入する。

この層の目的は、以下の4つである。

- ユーザー理解を、より感覚的で親しみやすい形で伝える
- 「まだ理解が育っていない状態」も魅力的に表現する
- ユーザーに愛着と継続利用の動機を与える
- SNS共有や会話のきっかけとなる、拡散性のある体験をつくる

---

## Core Principle

Character Expression Layerは、**理解そのもの**ではなく、**理解を表現するための層**である。

Compassは、UserModelを根拠付き仮説として慎重に扱う。

そのため、キャラクター表現をUserModel本体と混同してはならない。

### 原則

1. キャラクターはUserModel本体ではない
2. キャラクターはEvidenceおよび確認済み理解をもとに生成される
3. 根拠が不十分な段階では、完成形を出さず未成熟な姿で表現する
4. キャラクター表現を逆流させてUserModelを更新してはならない
5. キャラクターは参考的・娯楽的な表現であり、人格そのものではない
6. ユーザーが気に入らない場合、非表示・再生成・表現変更ができる余地を残す

---

## Relationship to UserModel

Compassにおける理解の正式な流れは、以下である。

```text
DailyLog / SleepRecord / その他観測データ
        │
        ▼
Analysis
        │
        ▼
Evidence
        │
        ▼
Understanding Candidate
        │
        ▼
User Confirmation
        │
        ▼
UserModel
```

Character Expression Layerは、この流れの後段に位置する。

```text
DailyLog / SleepRecord / その他観測データ
        ↓
Analysis
        ↓
Evidence
        ↓
Understanding Candidate
        ↓
User Confirmation
        ↓
UserModel
        ↓
Character Interpretation
        ↓
Character Expression State
        ↓
Character Visual Expression
        ↓
Share / Entertainment Experience
```

それぞれの位置付けは、以下のとおりである。

```text
UserModel
→ Compassが慎重に保持する理解の本体

Character
→ その理解を親しみやすく翻訳した表現
```

## Why This Matters

### 1. データ不足の初期状態を自然に扱える

使い始めた直後は、Compassはまだ十分な理解を持っていない。

その段階で完成されたキャラクターを出すと、AIがもっともらしく知ったかぶりをしているように見え、Compassの誠実さと矛盾する。

そこで、Character Expression Layerでは、理解が十分に育つまではキャラクターを**卵の状態**で表現する。

これにより、「まだ分からない」という状態をネガティブに見せず、むしろ「これから育っていく楽しみ」として提示できる。

### 2. Compassの「理解は育つ」という思想と一致する

Compassは、ユーザーを最初から断定的に理解するのではなく、記録と対話を重ねながら少しずつ理解を育てる。

```text
卵
 ↓
孵化
 ↓
成長
```

という構造は、この思想を非常に自然に表現できる。

### 3. 愛着と継続利用につながる

ユーザーは、自分の理解の成長を抽象的な数値だけで見るよりも、キャラクターの変化として見たほうが愛着を持ちやすい。

例:

- まだ卵のまま
- ついに孵化した
- 最近少し元気そうになった
- 前より落ち着いた雰囲気になった

こうした変化が、日々の記録を続ける動機になる。

### 4. 共有性と拡散力を持たせられる

キャラクターが生成されることで、ユーザー同士が以下のような行動を取りやすくなる。

- 自分のキャラクターを見せ合う
- キャラクターの違いや共通点を話題にする
- SNSでキャラクターや成長記録を共有する

これにより、Compassを利用していない人にも自然にサービスが認知される可能性がある。

## Character Growth Concept

Character Expression Layerでは、キャラクターを固定されたプロフィールとしてではなく、**育っていく存在**として扱う。

### 基本コンセプト

- 理解が不十分な間は「卵」として表現する
- 十分な記録と根拠が揃うと孵化する
- 継続的な理解に応じて見た目や雰囲気が変化する
- 変化は一方向ではなく、状態に応じて若返る・落ち着く・元気になるなど、可逆的でよい

### なぜ「卵」がよいか

- 初期段階のデータ不足を誠実に表現できる
- ユーザーに「まだ理解が育っていない」ことを自然に伝えられる
- 成長への期待感を生み、体験として魅力的になる
- 初期段階における不正確な人格表現を避けられる

---

## Character Stages

以下は、将来構想としての代表的な成長段階である。

### Stage 0: Egg

- まだ十分な理解がない状態
- Compassがユーザーを観測している段階
- キャラクターの個性はまだ確定していない
- 説明文は「Compassが少しずつあなたを理解しています」程度に留める
- 無理に性格やタイプを表現しない

### Stage 1: Hatching

- 最低限の観測データが集まり、いくつかの傾向が見え始めた状態
- 卵に模様や光、色などの変化が現れる
- キャラクターの一部が見え始める
- まだ仮説が多く、不安定な状態として扱う

### Stage 2: Growing

- 複数のEvidenceや確認済み理解が蓄積した状態
- キャラクターの見た目や個性がある程度明確になる
- 表情やしぐさにもユーザーの傾向が現れ始める
- 長期理解と短期状態の違いが、少しずつ外見へ反映される

### Stage 3: Mature

- 十分な理解が蓄積され、長期傾向と短期状態の両方が反映される状態
- キャラクターの外見・雰囲気・モチーフ・説明が比較的安定する
- 共有可能なキャラクター名や説明を持つことができる
- ただし、完成して固定された人格ではなく、今後も変化し得る存在として扱う

---

## Growth Conditions

キャラクターの成長段階は、単に利用日数だけで決めるのではなく、**データの量と質の両方**を基準にする。

### 成長判定に利用できる候補

- 記録日数
- DailyLogの件数
- SleepRecordの件数
- Evidenceの件数
- Understanding Candidateの件数
- User Confirmationの件数
- 確認済みUnderstandingの件数
- Evidence付きUserModel理解の件数
- 複数領域における観測の多様性
- 同じ傾向が一定期間繰り返し観測されているか

### 基本方針

- 日数だけで孵化させない
- 根拠が少ない状態で完成形を出さない
- 単に記録件数を増やすことを目的にしない
- 「見た目が派手になること」ではなく、「理解が育つこと」を成長基準にする
- ユーザーが無理に記録を増やさなければならない体験にしない

## Long-term and Short-term Expression

Character Expression Layerでは、長期理解と短期状態を分けてキャラクターへ反映する。

### 長期理解が反映されるもの

比較的安定した要素。

- 基本色
- キャラクターのモチーフ
- 服装や装飾の系統
- キャラクターの種族や世界観
- 全体的な雰囲気
- ベースとなる性格表現
- 持っている小物
- キャラクター名の方向性

### 短期状態が反映されるもの

比較的変化しやすい要素。

- 表情
- しぐさ
- 姿勢
- 背景演出
- 動きの軽さ・重さ
- エネルギー感
- 一時的なオーラ
- 光や天候の表現

### 例

```text
最近疲労が高い
→ 少し眠そうな表情、ゆっくりした動き

集中している
→ 落ち着いた視線、静かな雰囲気

気分が良い
→ 明るい表情、軽やかな動き

不安が強い
→ 少し身を縮める、守るようなしぐさ

睡眠が安定している
→ 表情がやわらかくなり、姿勢が開く
```

## Character Is Not Age in the Real-World Sense

キャラクターの変化を「年齢」に見える表現として扱う場合でも、それは現実の年齢のような一方向の時間経過とは限らない。

Compassにおいて、キャラクターの「若さ」「成熟」「落ち着き」は、以下のような象徴的な表現として扱う。

- 現在の理解の成熟度
- 現在の心身状態
- 行動傾向の変化
- 精神的な軽やかさ・重さ
- 活力や安心感
- 新しいことへ向かう姿勢
- 経験による落ち着き

そのため、以下のような変化も自然に起こり得る。

- 以前より若々しい印象になる
- 疲れた時期に少し老成した印象になる
- 安定してきたことで落ち着いた雰囲気になる
- 元気が戻ることで軽やかな姿になる
- 新しい目標を見つけたことで、好奇心の強い姿になる

キャラクターの変化は、現実の老化を再現するものではない。

ユーザーの現在の状態や変化を、象徴的に伝えるための表現である。

## Character Types as Entertainment Expression

将来的には、Compassが観測した傾向をもとに、

> あなたは現在、こんなキャラクターです。

と表現するエンターテインメント要素を導入できる。

例:

- 夜の観測者タイプ
- 風を読む集中航海士タイプ
- がんばりすぎる灯台守タイプ
- 静かな森の探究者タイプ
- 仲間を導く星明かりの案内人タイプ

これらは正式な診断名ではなく、**現在見られている傾向を表現したキャラクター名**である。

エンターテインメントとしては、ある程度分かりやすく、印象に残る表現を使用してよい。

ただし、理解の本体とは明確に分離する。

### 注意点

- UserModel本体にキャラクターのタイプ名をUnderstandingとして保存しない
- タイプ名はEvidenceの代わりにならない
- タイプ名を根拠として新しい理解を生成しない
- キャラクター結果からUserModelを更新しない
- あくまで現在見られている傾向を翻訳した表現とする
- ユーザーが「自分とは違う」と感じた場合、表現を調整できるようにする

## Visual Generation Concept

Character Expression Layerの大きな魅力の一つは、キャラクター画像の生成にある。

### 目指す方向

- ユーザーごとに異なるキャラクターを生成する
- 同じ傾向を持つ人は、ある程度似た系統の外見になる
- 同じ系統であっても、個人差が残る
- 長期理解と短期状態の両方を反映する
- ユーザーが共有したくなる見た目をつくる
- Compass全体として統一された世界観を維持する

### 反映可能な要素

- 基本色
- モチーフ
- 種族やシルエット
- 髪型や服装の方向性
- 表情
- 背景
- 小物
- 姿勢
- 雰囲気
- 光の演出
- 動きの軽さや重さ
- 季節や天候の表現

### 例

```text
睡眠不足の影響を受けやすい
→ 夜、月、柔らかい青、眠そうな目元

好奇心が高い
→ 地図、コンパス、望遠鏡、未知の光

一人での集中を好む
→ 静かな書斎、落ち着いた光、小さな机

回復傾向にある
→ 明るい色、開いた姿勢、朝日の背景

疲労が強い
→ 眠そうな表情、静かな背景、重い動き

人との関係を大切にする
→ 灯り、つながった星、仲間を守る装飾
```

---

## Shared Visual Language

同じような傾向を持つユーザーには、キャラクターの一部に共通した特徴を持たせることができる。

例:

- 睡眠や夜の影響を受けやすい人には、月や夜空のモチーフ
- 好奇心が強い人には、地図や望遠鏡のモチーフ
- 人間関係を大切にする人には、灯りやつながりを表すモチーフ
- 静かな環境で力を発揮する人には、森や書斎のモチーフ

これにより、ユーザー同士がキャラクターを見たときに、

- 自分と似た系統である
- 一部は似ているが、別の部分は違う
- 自分にはない特徴を持っている

と感じられる。

ただし、ユーザーを単一のタイプへ固定してはならない。

複数の傾向を組み合わせ、それぞれのユーザーに異なるキャラクターを生成する。

## Shareability

Character Expression Layerは、ユーザー同士の会話やSNS共有も視野に入れる。

### 共有しやすい要素

- 現在のキャラクター画像
- キャラクター名
- 現在の成長段階
- 最近の雰囲気の変化
- 卵からの成長記録
- 孵化した日
- 「今のあなたを一言で表すと」という短い説明
- キャラクターの主なモチーフ
- 友人との共通点や違い

### 期待できる効果

- 継続利用のきっかけ
- ユーザー同士の話題化
- SNSでの自然な拡散
- Compassらしさの認知拡大
- 自己理解について話すきっかけ
- 新しいユーザーがCompassへ興味を持つ入口

### 共有時の注意

共有カードには、プライバシーに関わるEvidenceや日記内容を直接含めない。

共有されるのは、ユーザーが許可したキャラクター画像・名前・短い説明などに限定する。

## Explainability

Compassでは、キャラクター表現も完全なブラックボックスにはしない。

必要に応じて、ユーザーが「なぜ今このような見た目なのか」を確認できるようにする。

### 例

- 最近、睡眠が不安定な記録が続いているため、少し疲れた表情になっています
- 集中に関する傾向が見られるため、静かな雰囲気が強くなっています
- 気分が安定してきたため、以前より明るい表現になっています
- 新しいことへの関心が増えたため、探検を表す小物が追加されました

ただし、説明しすぎてキャラクターの世界観を壊さないよう、詳細表示は任意で開ける形が望ましい。

```text
キャラクター
    ↓
「どうしてこの姿なの？」
    ↓
使用された理解や最近の変化を確認
```

## Safeguards

Character Expression Layerには魅力がある一方で、誤用の危険もある。

そのため、以下のガードレールを設ける。

### 禁止事項

1. 根拠不十分な状態で完成形のキャラクターを出す
2. キャラクター表現をUserModel本体として扱う
3. キャラクター名を正式な人格診断として使う
4. Character結果をそのままUserModelへ逆流させる
5. 共有されやすさを優先して、過激で否定的な表現に寄せる
6. 疲労・不安・落ち込みを、醜さや価値の低さとして表現する
7. キャラクターの見た目によってユーザーを評価する
8. 本人が望まないプライベート情報をキャラクターへ反映する

### 必須条件

1. CharacterはEvidenceまたは確認済みUnderstandingをもとに生成する
2. 初期段階は卵または未成熟な表現を使用する
3. 参考的・娯楽的な表現であることを明示する
4. ユーザーが嫌な場合、非表示または弱表示にできる
5. 再生成や一部要素の変更ができる余地を残す
6. 共有はユーザーの明示的な操作によってのみ行う
7. キャラクター表現からUserModelを更新しない

## Example User Experience

### 初期

- キャラクターは卵
- 「Compassが少しずつあなたを理解しています」と表示する
- まだ人格やタイプに関する説明は行わない
- 記録が増えることで卵の色や模様が少しずつ変わる

### 孵化前

- 卵に小さなひびや光が現れる
- 「いくつかの傾向が見え始めています」と伝える
- 孵化までの正確な日数は断定しない
- 記録を強制するような表現は行わない

### 孵化

- キャラクターが初めて姿を現す
- 最初のキャラクター名や短い説明が提示される
- 「現在の記録から見えている姿です」と明示する
- ユーザーが感想や違和感を伝えられる

### 継続利用後

- キャラクターに個性が現れる
- 気分や疲労によって表情や雰囲気が変わる
- 新しい理解に応じて小物やモチーフが追加される
- ユーザーが「前より落ち着いた感じになった」と変化を実感できる
- 過去の姿と現在の姿を比較できる

## Suggested Data Model Direction

あくまで将来案として、以下のような派生表現モデルを持つことが考えられる。

```typescript
type CharacterStage =
  | 'EGG'
  | 'HATCHING'
  | 'GROWING'
  | 'MATURE';

type CharacterMoodExpression =
  | 'BRIGHT'
  | 'CALM'
  | 'TIRED'
  | 'ANXIOUS'
  | 'FOCUSED';

type CharacterEnergyExpression =
  | 'LOW'
  | 'MEDIUM'
  | 'HIGH';

type CharacterVisualMaturity =
  | 'YOUNG'
  | 'BALANCED'
  | 'MATURE';

interface CharacterExpressionState {
  stage: CharacterStage;

  visualMaturity: CharacterVisualMaturity;

  moodExpression: CharacterMoodExpression;

  energyExpression: CharacterEnergyExpression;

  motifs: string[];

  styleTraits: string[];

  summaryLabel?: string;

  sourceUnderstandingIds: string[];

  createdAt: string;

  updatedAt: string;
}
```

これは正式なTypeScript型ではなく、将来の検討材料である。

このモデル方向性について、現時点では以下を明確な前提とする。

- 現在未実装である
- UserModel本体とは別管理である
- アプリケーションコード上の正式なTypeScript型ではない
- 実装前にADRと詳細設計が必要である
- `sourceUnderstandingIds` は説明可能性のための参照であり、キャラクター表現からUserModelを更新するためのものではない

- 誠実さ

- 自己理解の実感
- 共有しやすさ
- サービスとしての魅力

Compassは、ユーザーを早く断定するのではなく、理解が育っていく過程そのものを体験にする。

Character Expression Layerは、その思想を最も分かりやすく、魅力的に可視化するための重要な候補である。

- 理解が育ると孵化・成長する
- 長期理解と短期状態が見た目に反映される
- 成長や変化は一方向ではなく、可逆的である
- ユーザー同士で共有しやすい体験をつくる
- キャラクター表現とUserModel本体を明確に切り分ける
- キャラクターの結果を理解の本体へ逆流させない

これにより、Compassは「理解の正確さ」と「体験としての楽しさ」の両立を目指す。

Compassが理解不足を隠すのではなく、卵から育っていく存在として表現することで、ユーザーは理解が形成される過程そのものを楽しめる。
